# 虚拟情感交流机器人系统（A22）项目详细方案

## 1. 项目概述
本项目面向“服务外包创新创业大赛 A22 赛题”，目标是建设一套可部署、可演示、可交付的虚拟情感交流机器人系统。系统基于“本地交互 + 远端推理”架构，支持文本/语音输入，输出数字人语音视频回复，形成完整情感交互闭环。

---

## 2. 建设目标与指标
### 2.1 建设目标
1. 实现用户输入到数字人回复的端到端流程。
2. 保证系统可复现启动、可脚本化运维。
3. 提供可比赛展示的稳定交互体验。

### 2.2 核心指标（建议）
1. 可用性：核心链路成功率 >= 95%（演示场景）。
2. 时延：首个可播放视频片段在可接受范围内（与硬件环境相关）。
3. 稳定性：连续多轮对话无服务异常退出。

---

## 3. 总体架构设计
系统由三层组成：

1. 交互层（本地）
- Web 前端：聊天、录音、状态显示、数字人视频展示。
- 本地 edge-backend：与远端编排服务对接、接口统一。

2. 编排层（远端）
- Orchestrator：统一接收请求，完成多服务调度、策略拼装与结果聚合。

3. 模型服务层（远端）
- qwen-server：LLM 推理服务。
- speech-service：语音识别及语音相关特征处理。
- vision-service：视觉理解与情绪线索提取。
- avatar-service：TTS + 数字人视频生成。

数据主链路：  
`用户输入 -> Orchestrator -> (LLM / Speech / Vision / Avatar) -> 前端展示`

---

## 4. 关键功能模块设计
### 4.1 对话理解与生成模块
1. 输入模式：text / audio（可扩展 video）。
2. 能力：多轮上下文拼接、情绪风格映射、策略化回复生成。
3. 输出：结构化回复对象（reply_text / emotion_style / avatar_action / media url）。

### 4.2 语音识别模块（ASR）
1. 将用户语音转为文字输入编排层。
2. 输出识别文本及可选置信信息，作为后续推理依据。

### 4.3 语音合成模块（TTS）
1. 使用 CosyVoice-300M-Instruct 生成中文回复语音。
2. 将结果保存为 wav/base64，供前端或数字人渲染链路使用。

### 4.4 数字人驱动模块
1. 使用 SoulX-FlashHead 进行人脸驱动与视频生成。
2. 支持分段视频清单（manifest）与完整视频路径输出。
3. 通过参考图驱动不同数字人形象（可配置多 profile）。

### 4.5 前端交互模块
1. 聊天窗口：展示用户消息与数字人回复。
2. 语音输入：录音状态提示、发送控制。
3. 数字人面板：头像/视频展示、状态信息展示、形象切换。

## 5. 运行环境与依赖
### 5.1 远端环境
1. Linux + GPU 运行环境。
2. tmux 管理多服务进程。
3. 模型目录统一挂载在 `/root/autodl-tmp/a22/models`。

### 5.2 本地环境
1. Windows + Docker Desktop。
2. 通过 compose 启动 `frontend` 与 `edge-backend`。
3. SSH 隧道将本地代理到远端 orchestrator。

---

## 6. 端到端流程说明
### 6.1 文本输入流程
1. 前端提交文本到 edge-backend。
2. edge-backend 转发到远端 orchestrator `/chat`。
3. orchestrator 调用 LLM 生成回复文本。
4. avatar-service 对回复文本做 TTS，并调用 SoulX 生成视频。
5. 返回文本 + 音频 + 视频地址，前端渲染并播放。

### 6.2 语音输入流程
1. 前端采集音频上传。
2. speech-service 完成 ASR 文本转写。
3. 其余流程与文本输入一致，返回多模态回复。

## 7. 配置与部署方案
### 7.1 远端服务启动
采用脚本化方式启动/停止：
- `scripts/remote/start_remote_stack_tmux.sh`
- `scripts/remote/stop_remote_stack_tmux.sh`

健康检查接口：
- `http://127.0.0.1:19000/health`（orchestrator）
- `http://127.0.0.1:19300/health`（avatar-service）

### 7.2 本地服务启动
使用 Docker Compose：
- `compose.yaml`
- `compose.local.yaml`

启动命令：
- `docker compose -f compose.yaml -f compose.local.yaml up -d frontend edge-backend`

---

## 8. 可靠性与风险控制
### 8.1 可靠性措施
1. 服务分层与独立进程，便于故障隔离。
2. 启动脚本统一环境变量，减少人工误配。
3. 健康检查与日志排障路径明确。

### 8.2 主要风险与对策
1. 模型加载失败：提供健康检查与日志快速定位。
2. 网络不稳定：采用隧道保活参数与重连机制。
3. 音视频播放异常：前端与后端分层排查，保证可回退配置。
