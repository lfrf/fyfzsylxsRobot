# 桌面情感交互机器人 V1 总体设计方案（可评审版）

## 1. 方案定位与目标

### 1.1 项目定位
本方案将现有 `FuChuangSai_A22` 语音对话能力从“数字人渲染终端”改造为“桌面机器人终端”。  
机器人侧使用树莓派完成设备控制与本地交互，复杂推理继续放在远程算力侧，以降低本地硬件成本和调试难度。

### 1.2 V1 总目标
1. 实现稳定的人机语音对话（近场 0.5m 到 1.5m）。
2. 实现机器人头部 2 自由度动作（水平转头 + 俯仰点头）。
3. 实现双 OLED “眼睛”表情反馈。
4. 保留现有远程服务启动链路，减少对原有后端能力改动。
5. 形成可复制装配、可验收、可演示的完整工程包。

### 1.3 设计原则
1. 先可用后优化：V1 优先稳定性与可维护性。
2. 软硬件解耦：头部控制、显示、语音链路可独立替换。
3. 安全优先：舵机电源与主控电源分离，保留急停机制。
4. 可迭代：预留声源定位、唤醒词、机身一体化音频升级接口。

## 2. 功能范围

### 2.1 V1 范围内功能
1. 文本对话到语音播报闭环。
2. 摄像头输入与视觉注意力驱动头部动作。
3. 双 OLED 基础表情（注视、眨眼、说话态）。
4. 系统健康检查与一键启动脚本。

### 2.2 V1 暂不纳入功能
1. 底盘移动与导航。
2. 机械手抓取。
3. 高精度多说话人声源定位（DoA）闭环。
4. 本地大模型离线推理。

## 3. 总体系统架构

### 3.1 架构分层
1. 机器人本体层：树莓派、云台舵机、双 OLED、麦克风、扬声器、摄像头、电源模块。
2. 本地服务层：`edge-backend`、`robot-runtime`。
3. 远程推理层：`orchestrator`、`qwen-server`、`speech-service`、`vision-service` 等。

### 3.2 数据流
1. 用户语音进入麦克风。
2. 本地服务将请求转发到远端推理。
3. 远端返回文本与动作意图。
4. 本地触发语音播报、眼睛表情、头部动作。

### 3.3 关键服务关系
1. `robot-runtime` 负责机器人设备控制与动作映射。
2. `edge-backend` 负责协议归一化与远程转发。
3. `scripts/remote/start_remote_stack_tmux.sh` 负责远端主链路拉起。

## 5. 结构设计方案

### 5.1 最终尺寸建议（V1）
1. 整机外形：`170mm(W) × 150mm(D) × 230mm(H)`。
2. 头部外壳：`145mm × 120mm × 105mm`。
3. 底座外壳：`170mm × 150mm × 45mm`。
4. 云台安装腔：`90mm × 80mm × 70mm`（内部净空）。

### 5.2 三层结构布局
1. 头部层：双 OLED、摄像头、麦克风前开孔、轻量外壳。
2. 中轴层：DS3218 双轴机构、颈部线束通道、限位保护。
3. 底座层：树莓派、电源转换、PCA9685、功放与布线固定。

### 5.3 重心与稳定性要求
1. 整机重心尽量保持在底座中心附近。
2. 头部载荷建议控制在 300g 内，减轻舵机冲击。
3. 底座可加钢配重片，提高转头时抗倾覆能力。

### 5.4 壳体与打印建议
1. 外壳材料建议 PETG，壁厚 2.4mm。
2. 结构连接点建议使用热熔铜螺母，便于反复拆装。
3. 舵机固定板建议高填充（40% 及以上）并增加筋位。

## 6. 云台与舵机设计

### 6.1 选型理由
1. DS3218 20kg 比微型舵机余量更高，适合“机器人头部”刚性结构。
2. 180° + 270°组合可以平衡控制边界与可动范围。
3. 与 PCA9685 配合成熟，维护成本低。

### 6.2 动作边界建议
1. 水平轴（Pan）：中心左右各约 70°。
2. 俯仰轴（Tilt）：下俯约 25°，上仰约 35°。
3. 软件限速：每 20ms 最大变化约 2°，减小抖动与电流峰值。

### 6.3 扭矩校核方法
1. 估算公式：`T(kg·cm) = 头部质量(kg) × 重心半径(cm) × 安全系数(2~3)`。
2. 示例：`0.35kg × 5cm × 3 = 5.25kg·cm`，20kg 舵机余量充足。

## 7. 电气系统与供电设计

### 7.1 电源架构
1. 舵机与树莓派分轨供电。
2. 舵机电源：`6V 8A` 独立输出到 PCA9685 `V+`。
3. 树莓派电源：官方适配器独立供电。
4. 所有模块必须共地。

### 7.2 接线规范（关键）
1. Pi `GPIO2/GPIO3` 接 PCA9685 的 `SDA/SCL`。
2. Pi `3.3V` 接 PCA9685 `VCC`（逻辑电），Pi `GND` 接 PCA9685 `GND`。
3. DS3218 水平舵机接 `CH0`，俯仰舵机接 `CH1`。
4. 舵机电源正负极接 PCA9685 舵机电源端子，不走 Pi 5V。
5. SPI OLED 与相机线束避开舵机供电线，减少干扰。

### 7.3 SPI OLED 推荐引脚（示例）
1. 共用 `SCLK(GPIO11)`、`MOSI(GPIO10)`、`RST(GPIO27)`。
2. 左眼：`CS(GPIO8)`、`DC(GPIO25)`。
3. 右眼：`CS(GPIO7)`、`DC(GPIO24)`。

### 7.4 安全与保护
1. 6V 舵机支路串联保险丝与总开关。
2. 舵机电源入口建议并联 1000uF 电容抑制压降。
3. 预留硬件急停按钮，触发后切断舵机供电。

## 8. 音频方案设计

### 8.1 当前推荐
1. 麦克风：`reSpeaker Lite`（2 麦，USB）。
2. 外放：开发期先用 USB 音箱，后续可切换机身内置功放方案。

### 8.2 为什么可行
1. 成本可控，满足你当前预算。
2. 具备 AEC/AGC/NS 等语音前端能力，满足近场交互。
3. USB 接入调试快，先保证系统跑通。

### 8.3 安装建议
1. 麦克风尽量放头部前侧，远离舵机与电源模块。
2. 扬声器尽量向前出声，避免直冲麦克风。
3. 麦克风与扬声器物理间距建议大于 8cm。

## 9. 软件架构与部署方案

### 9.1 代码与配置位置
1. 机器人运行时：`local/robot-runtime`。
2. 机器人组合编排：`compose.robot.yaml`。
3. 远程启动脚本：`scripts/remote/start_remote_stack_tmux.sh`。

### 9.2 本地服务职责
1. `edge-backend`：转发与协议统一。
2. `robot-runtime`：将回复映射为眼睛、头部、音频动作。
3. `robot-runtime` 当前默认 `mock`，逐步替换为真实硬件驱动。

### 9.3 服务启动（本地）
```bash
docker compose -f compose.yaml -f compose.robot.yaml up -d edge-backend robot-runtime
```

### 9.4 服务启动（远端）
```bash
cd /root/autodl-tmp/a22/code/FuChuangSai_A22
./scripts/remote/stop_remote_stack_tmux.sh
./scripts/remote/start_remote_stack_tmux.sh
```

### 9.5 健康检查
```bash
curl -s http://127.0.0.1:8000/health
curl -s http://127.0.0.1:18100/health
```

## 10. 控制策略与时延目标

### 10.1 V1 控制策略
1. 视觉优先：由摄像头跟踪触发头部朝向。
2. 语音链路：本地采集，远端识别与生成。
3. 表情驱动：按回复语义映射至眼睛动画与头部细动作。

### 10.2 建议时延指标
1. 语音输入到开始回复：`< 1.5s`（网络正常条件）。
2. 头部动作响应：`< 200ms`（本地控制环）。
3. OLED 表情切换：`< 100ms`。

## 11. 装配与调试流程

### 11.1 装配步骤
1. 先完成底座电源与主控安装。
2. 安装 PCA9685 与双舵机机构，空载测试。
3. 安装头部壳体、相机、双 OLED、麦克风。
4. 完成线束整理与应力释放。
5. 执行软件联调与动作边界校准。

### 11.2 调试顺序
1. 仅上电树莓派，检查系统稳定。
2. 单独上电舵机支路，校验转动范围。
3. 打通摄像头、麦克风、音箱。
4. 执行端到端对话与动作测试。

### 11.3 验收检查项
1. 连续运行 30 分钟无掉线或异常重启。
2. 连续执行 200 次转头动作无卡滞。
3. 1m 距离语音识别准确率达到可演示水平。
4. 现场演示时对话、转头、眼睛表情协同稳定。

## 12. 可靠性与风险控制

### 12.1 主要风险
1. 舵机电流冲击导致主控掉压。
2. 扬声器回声影响语音识别。
3. 线束在颈部反复摆动造成接触不良。
4. 网络抖动导致回复时延波动。

### 12.2 对策
1. 电源分轨并加电容与保险保护。
2. 增加 AEC 参数调优与物理隔音设计。
3. 颈部线束使用蛇形余量与软固定夹。
4. `robot-runtime` 增加超时降级与重试。
